# 长时程智能体工作流

## 工作中综合

长时程智能体工作流（long-running agent workflows）的关键不是单纯扩大上下文窗口，而是让每个会话或智能体阶段都能安全交接。Anthropic 的早期模式把首次运行视为环境初始化，把之后每次运行限制为一个可验证增量；后续模式在瓶颈从连续性转向范围界定、判断和 QA 时，引入 planner/generator/evaluator 分离。

## 来源支持的模式

- **首次提示词不同**：initializer agent 先搭脚手架，而不是直接做功能。
- **状态外化**：`claude-progress.txt`、git history、feature list 和 `init.sh` 组成跨会话连续性层。
- **一次只做一个功能**：coding agent 选择一个失败或高优先级 feature，而不是试图一次性完成整个应用。
- **干净状态交接**：每个会话结束时都应留下可合并状态、描述性 commit 和进展更新。
- **用户视角验证**：Web app 功能要通过浏览器自动化验证，而不只是读代码或跑狭窄单测。
- **近似不可变的需求**：feature list 是契约；agent 可以在验证后标记通过/失败，但不应弱化测试描述。
- **升级为 planner/generator/evaluator**：复杂应用中，planner 扩展需求，generator 构建，evaluator 提供外部 QA。
- **Sprint contract**：当产品规格有意保持高层时，generator 与 evaluator 可以在编码前协商“完成定义”。
- **模型相关的简化**：context reset、sprint 分解和 evaluator 频率都应随模型能力变化重新测试。

- **文件系统是长任务底座**：LangChain Anatomy 进一步说明，长任务会产生大量 token，filesystem + git 可以持久记录进展、支持 rollback/branch，并成为人类与多个 agent 的共享账本。
- **Ralph Loop 是另一种继续机制**：不同于 Anthropic 的显式 context reset / progress handoff，Ralph Loop 通过 hook 拦截退出并在 clean context 中继续对照目标工作。
- **Planning + self-verification 组合**：长时程不是只“继续运行”，还需要计划拆解、每步验证、失败回传和修复循环。
- **Claude Code shorthand 来源补充操作层**：tmux 保持长命令和日志可观察，git worktrees 隔离并行 Claude，`/fork` 用于非重叠任务，statusline 显示 context/model/todos 等运行状态。
- **Dynamic workflows 把长任务拆成按需生成的多智能体 harness**：Claude Code 可为当前任务创建 JavaScript workflow，分发子智能体、选择模型/worktree、做对抗验证或循环直到停止条件满足。
- **长任务失败模式需要结构性对策**：dynamic workflows 来源显式命名 agentic laziness、self-preferential bias 和 goal drift；这些问题分别对应 completion evidence、external evaluator、file-backed state / clean context 等 harness 防线。

## 开放问题

- 长时程工作应该使用一个通用 coding agent，还是拆出 planner、tester、cleanup 等专门智能体？
- initializer 的输出哪些应该一次性生成，哪些应该持续重新规划？
- 这些模式能否从 Web 应用开发迁移到科学研究、金融建模等任务？
- harness 能否动态判断何时值得为外部 evaluator 付出成本？
- dynamic workflow 如何在“继续直到完成”和 token/资源预算之间设置可审计停止条件？
