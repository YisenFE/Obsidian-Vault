# 智能体优先软件开发

## 工作中综合

智能体优先软件开发（Agent-first software development）不是“开发者多用一点 AI 补全”，而是把工程重心转向设计让智能体能安全行动、观察、验证、恢复和持续改进的系统。OpenAI 更强调仓库与运行时环境；Anthropic 更强调跨会话交接和专门角色；Böckeler 更强调用户侧的生成前引导（feedforward guides）与生成后反馈检测器（feedback sensors）；LangChain 更强调基于执行轨迹（trace）的 harness 实验和中间件/调用拦截层（middleware）；NLAH 论文更强调把 harness policy 外化为可执行、可审计、可消融的自然语言对象；Michael Bolin 的 Codex 访谈则强调 agent loop、sandbox、少量强工具和多 workstream 的 developer inner loop；Claude Code shorthand 来源补上了个人 operating harness 的具体配置层。

## 来源支持的模式

- **人类掌舵，智能体执行**：人类负责排序、指定目标、验收和编码判断；智能体负责实现、处理反馈和推进变更。
- **开发者内循环转向调度与优化**：当多个 agent workstreams 并行推进时，人类要管理任务切分、差异审阅、技能沉淀、团队共享和上下文切换，而不只是连续手写代码。
- **代码库成为运行环境**：文档、计划、架构约束和质量状态进入仓库，让智能体可发现、可读取、可执行。
- **生成前引导优先**：`AGENTS.md`、skills、架构文档、语言服务器、CLI 脚本、codemods 和本地上下文注入会提高首次生成正确率。
- **反馈循环优先于一次性提示词**：UI 自动化、可观测性查询、测试、review agent、静态分析、执行轨迹和 CI 让智能体能自修正，也让 harness 工程师能调优外循环。
- **把品味机械化**：架构、风格偏好和设计标准应沉淀为 lint、结构测试、评估 rubric 或其他检测器，而不是只停留在人类临时评论中。
- **管理智能体熵**：后台清理任务像垃圾回收一样，持续清除 AI 生成代码带来的漂移和残留。
- **跨会话交接**：长时程工作需要进展记录、git history、feature contract 和启动脚本，保证新上下文能安全接续。
- **专门角色分工**：planner/generator/evaluator 分离可以缓解一个智能体在范围界定、自我评价和外部 QA 上的弱点。
- **中间件作为护栏**：pre-completion checklist、上下文注入、循环检测和时间预算提醒，都是对当前模型失败模式的补偿。
- **少量强工具 + 沙箱边界**：Codex 访谈显示，另一种产品侧取向是不给 agent 过多专用 wrapper，而是提供 terminal 这类强通用工具，并用 sandbox、权限策略和输出写文件等机制控制风险与上下文污染。
- **脚手架依赖模型与代码库**：随着模型能力提升、代码库可 harness 化程度变化、benchmark trace 暴露新失败模式，harness 复杂度需要重新评估。
- **可执行自然语言 policy**：NLAH/IHR 说明 harness policy 可以从 code controller 中外化出来，但 parser、sandbox、tool execution、validator 等精确机制仍应留在代码层。

- **Agent = Model + Harness**：LangChain Anatomy 把智能体优先开发的底层公式压缩为模型智能加外部工作支架；这让 filesystem、bash、sandbox、context management、verification 都变成一等工程对象。
- **模型与支架共同演化**：有用的 harness primitive 会进入产品和后训练循环；团队既要利用模型对工具协议的熟悉度，也要警惕对特定 harness 的过拟合。
- **个人操作支架也需要预算管理**：Claude Code shorthand 显示，skills、hooks、subagents、MCPs、plugins、worktrees、tmux、editor integration 都能提升能力，但必须控制启用工具数量和上下文成本。
- **Goal 把完成定义变成线程级状态**：Using Goals in Codex 显示，长任务不是靠用户不断说“继续”，而是靠 thread-scoped completion contract、verification surface、continuation gates 和 budget accounting。

## 开放问题

- 当人类仍然手写部分代码时，哪些智能体优先实践仍然成立？
- 衡量智能体优先开发健康度时，应该优先看周期时间、回滚率、缺陷密度、文档新鲜度、不变量违规数、交接质量、harness 覆盖率，还是 benchmark 增量？
- 团队如何在积极使用执行轨迹调优 harness 的同时，避免对 benchmark 或历史失败样本过拟合？
